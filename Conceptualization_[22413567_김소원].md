# 1. Business purpose
## 1.1. Development Background
>ADHD(Attention Deficit/Hyperactivity Disorder, 주의력결핍 및 과잉행동장애)란 주의력 결핍, 충동성 및 과잉행동의 주요 증상을 특징으로 하는 정신과적 장애이다.
> ADHD를 겪는 사람들은 시간 관리에 미숙하며 업무를 끝까지 집중력 있게 완료하지 못한다. 무엇보다 하기 싫은 과제 · 업무를 시작하기까지 오랜 시간이 걸리고 불편한 기분을 겪는다는 특징이 있다.

본인 또한 미약한 성인 ADHD 증상이 있는 사람으로서, 과제나 공부를 시작하는 것이 힘들어 불편함을 겪었다. 

이러한 불편함을 해소하기 위해 시장에 있는 여러 학습 관리 프로그램들을 사용해 보았으나, **대부분은 ‘과제의 양’보다 ‘공부 시간’을 중시했다.** 프로그램 사용 초반엔 집중이 쉬우나 몇십 분이 지나고 나면 집중력이 저하된 채 타이머를 켜놓고 멍때리거나 의미 없는 시간을 보내게 되었고, 결국 업무 효율이 저하되는 일이 빈번했다. 

과제 · 업무를 미루지 않고 시작할 수 있게 도와주고, 업무 시간이 아닌 업무 달성률에 집중하는 프로그램이 있으면 유용할 것 같아 **웹 앱, ‘원 스텝 천 리’ 를 개발**하게 되었다.
<br />
<br />
## 1.2. Development Goal
프로그램 개발 목적은 다음과 같다. 
<br />
<br />

>**1. 과제를 미루지 않도록 독려한다.**

이때 사용하는 방법은 브라이언 트레이시의  **'Eat that frog' 시간 관리법**이다. 해야 할 과제를 아주 작은 수준으로 나눠 심리적 거부감을 낮추는 것이다. 

‘보고서 5p 작성’은 힘들지만, ‘의자에 앉아 노트북 켜기’, ‘한글 파일을 켠 후 관련 논문 찾아보기’, ‘초안을 작성하기’, ‘수정하기’는 상대적으로 쉽다. 해당 관리법을 이용해 과제 시작에 대한 부담감을 낮추는 것이 첫 번째 목표이다. 또한 친구 추가 기능을 통해 사용자의 흥미도를 높여, 업무를 시작하는 것에 대한 거부감을 줄일 예정이다.

[사진 1-1. eat that frog]
<br />
<br />
<br />

>**2. 과제 달성도에 집중한다.**

오랫동안 과제를 붙잡고 있는다고 좋은 결과물이 나오는 것은 아니며, 시간은 한정되어 있다. 앞서 잘게 나눈 과제(하위 과제)들을 체크박스 형식으로 표시하고 하위 과제를 모두 달성하는 것이 핵심이다. 

이때 집중력을 높일 수 있도록 **포모도로 기법**(Pomodoro Technique) 타이머 기능을 제공한다. 포모도로 기법이란 경영 컨설턴트인 프란체스코 치릴로(Francesco Cirillo)가 제안한 시간 관리 방법론으로, **‘25분 집중 +　5분 휴식’ 사이클을 네 번 반복**한 후 30분간 휴식 시간을 갖는 관리법이다. 

체크박스를 모두 채우는 것이 주된 목표이며 포모도로 기법은 추가적으로 제공하는 기능임을 명확하게 밝힌다.

## 1.3. Target Market  
타겟 시장은 다음과 같다. 
- 시간 관리가 안 되는 ADHD 환자.
- ADHD 판정을 받지 않았더라도 미루는 습관이 있는 일반인들.
- 기존의 ‘공부 시간’ 중시 프로그램을 사용했을 때 효율성을 느끼지 못하는 사람들.

# 4. Concept of operation
해당 장에서는 앞서 기술했던 Use cases들이 어떻게 작동하는지 상세히 설명한다.

1. Register Member (회원가입)

| Purpose | 신규 사용자를 등록하고 개인 고유 코드를 생성한다. | 
| --- | --- | 
| Approach | 입력받은 아이디와 비밀번호를 서버로 전송해 사용자를 회원으로 등록하고 이러한 정보들을 테이블 형식으로 저장한다. 이때 가입에 사용한 아이디는 향후 친구 추가 기능에 사용하는 고유 식별자로 활용되므로 서버를 조회해 아이디 중복 여부를 확인해야 한다. | 
| Dynamics | 프로그램이 지원하는 기능을 사용하고자 본인의 계정을 만들고자 하는 경우. | 
| Goals | 사용자별 데이터를 저장한다. | 

<br />
<br />

2. Login (로그인)

| Purpose | 등록된 사용자인지 확인하고 서비스를 시작한다. | 
| --- | --- | 
| Approach | 사용자가 입력한 아이디와 비밀번호를 서버로 전송하여 저장된 정보와 대조해 인증을 수행한다. 인증에 성공하면 사용자의 기기에 인증 상태를 저장하여, 앱을 껐다 켜더라도 계속해서 로그인 상태를 유지할 수 있도록 처리한다. | 
| Dynamics | 본인의 과제를 관리하는 등 프로그램의 기능을 사용하기 위해 접속하는 경우. | 
| Goals | 개인화된 데이터에 접근할 수 있는 권한을 부여하고 서비스를 제공한다. | 

<br />
<br />

3. Logout (로그아웃)

| Purpose | 현재 사용 중인 서비스를 종료하고 데이터를 서버에 동기화한다. | 
| --- | --- | 
| Approach | 사용자가 로그아웃을 요청할 경우, 현재 데이터(종료되지 않은 타이머 시간, 체크된 과제 등)를 서버에 전송해 최종 업데이트한다. 동기화가 안전하게 완료되면 기기에 저장된 인증 정보를 삭제하여 접속을 해제한다. | 
| Dynamics | 다른 기기에서 접속하거나 개인정보 보호를 위해 사용 종료를 원할 때. | 
| Goals | 프로그램을 안전하게 종료해 타인의 데이터 접근을 차단하고 최신 상태를 저장한다. | 

<br />
<br />

4. Add Task (과제 추가)

| Purpose | 새로운 메인 과제 및 하위 과제를 생성한다. | 
| --- | --- | 
| Approach | 사용자가 과제 내용을 작성하면, 서버에 tasks 테이블을 만들고 데이터를 저장한다. 하위 과제는 메인 과제에 종속되도록 연결 관계를 설정하여 서버의 테이블에 체계적으로 기록한다. | 
| Dynamics | 사용자가 오늘 또는 향후에 해야 할 업무나 공부 계획을 세울 때. | 
| Goals | 해야 할 일을 체계적으로 분류해 목표를 명확히 관리할 수 있게 한다. | 

<br />
<br />

5. Manage Task (과제 관리)

| Purpose | 과제의 완료 또는 미완료 상태를 표시하고 서버에 갱신한다. | 
| --- | --- | 
| Approach | 사용자가 체크박스를 누르면 즉각적으로 화면의 시각적 상태를 변경하여 반응성을 높이고, 해당 과제의 변경된 상태 값을 서버로 전송, 저장한다. | 
| Dynamics | 할 일을 마쳤거나, 실수로 체크한 것을 취소하고자 할 때. | 
| Goals | 시각적인 성취감을 제공하고, 일일 달성률 계산을 위한 데이터를 기록한다. | 

<br />
<br />

6. Reminder Task (과제 리마인더)

| Purpose | 설정한 주기마다 미완료 과제 수행을 촉구하는 푸쉬 알림을 보낸다. | 
| --- | --- | 
| Approach | 서버에서 주기적으로 실행되는 자동화 작업을 설정한다. 서버의 데이터를 조회해 알림 설정이 켜져 있고 아직 완료하지 않은 과제가 있는 사용자를 찾아낸 뒤, 해당 사용자의 기기로 푸쉬 알림을 전송한다. 사용자는 푸쉬 알림을 몇 시간마다 받을지 설정할 수 있다. | 
| Dynamics | 사용자가 나태해지는 것을 방지하고 지속적으로 할 일을 상기시키고자 할 때. | 
| Goals | 사용자의 프로그램 사용을 독려하고 과제 완수를 유도한다. | 

<br />
<br />

7. Pomodoro Timer (포모도로 타이머)

| Purpose | 과제 수행 시 집중 시간을 측정하고 만료 시 알림을 제공한다. | 
| --- | --- | 
| Approach | 타이머 시작 버튼을 누른 시점의 시간을 기록하고, 시스템 시간을 기준으로 남은 시간을 독립적으로 계산한다. 종료 시 누적된 집중 시간을 서버로 전송해 저장하며, 설정된 시간이 끝나면 기기에 알림을 띄운다. | 
| Dynamics | 집중력을 올려 효율적으로 과제를 마칠 수 있도록 한다. | 
| Goals | 집중 시간을 측정한다. | 

<br />
<br />

8. Daily Report (데일리 리포트 보고)

| Purpose | 하루를 마감하며 오늘의 과제 달성률과 집중 시간을 계산해 알려주고, 목표 연속 달성 시 격려한다. | 
| --- | --- | 
| Approach | 매일 자정 직전, 서버에서 자동으로 오늘 생성된 과제 수와 완료 수를 비교해 달성률을 계산하고, 타이머 기록을 합산해 통계를 저장한다. 이때 달성률이 100%라면 사용자의 '연속 달성일' 수치를 증가시킨다. 계산된 통계 요약본과, 연속 달성 시 격려 메시지를 포함해 기기로 푸쉬 알림을 발송한다. | 
| Dynamics | 하루 일과가 끝나가는 시점에 오늘의 성과를 피드백받고 동기를 부여받고자 할 때. | 
| Goals | 사용자의 성취감을 고취시키고 자신의 패턴을 돌아보게 한다. | 

<br />
<br />

9. View Status (현황 확인)

| Purpose | 가입일 이후의 누적 과제 달성률과 연속 달성 기록을 조회한다. | 
| --- | --- | 
| Approach | 매일 통계 산출 시 서버에 미리 누적해서 저장해 둔 요약 통계 데이터와 연속 달성일 정보를 불러와 화면에 보여준다. | 
| Dynamics | 자신의 누적 통계를 시각적으로 확인하고 싶을 때. | 
| Goals | 장기적인 관점에서의 성과를 파악하게 하여 지속적인 목표 달성을 돕는다.  | 

<br />
<br />

10. Add Friend (친구 추가)

| Purpose | 개인 아이디를 이용해 다른 사용자를 친구 목록에 등록하고 연결한다. | 
| --- | --- | 
| Approach | 화면에서 친구의 아이디를 입력하면 서버에서 해당 사용자를 검색한다. 일치하는 사용자가 존재할 경우, 서버의 테이블에 두 사용자의 식별자를 묶어 기록함으로써 서로 친구 관계가 되었음을 저장한다. 이 때, 친구가 한 명 이상이 되면 해당 사용자를 ‘리더’로 등록한다. | 
| Dynamics | 지인과 함께 목표를 달성하기 위해 서로의 아이디를 공유하고 등록할 때. | 
| Goals | 사용자 간의 상호 감시 및 응원 환경을 조성함으로써 동기를 부여한다. | 

<br />
<br />

11. Register Joint Task (공동 과제 등록)

| Purpose | 친구들에게 공통된 과제를 일괄적으로 부여한다. | 
| --- | --- | 
| Approach | 친구가 한 명 이상인 사용자는 모두 리더이며, 리더는 공동 과제를 추가할 수 있다. 사용자(리더)가 과제를 작성한 후 공동 과제로 등록하면, 서버는 해당 사용자의 친구 목록을 조회한다. 그 후 각 친구들의 과제 목록 테이블과 본인의 과제 목록 테이블에 동일한 내용의 과제를 일괄적으로 복사하여 저장한다. 이를 통해 각자는 개인 숙제처럼 과제를 독립적으로 체크하고 관리할 수 있다. | 
| Dynamics | 스터디 모임의 진도 체크나 팀 프로젝트의 공통 할 일을 그룹원 모두에게 부여할 때. | 
| Goals | 여러 명에게 동일한 과제를 일일이 등록하는 수고를 덜고 그룹 목표를 효율적으로 설정한다. | 

<br />
<br />

12. View Ranking (랭킹 확인)

| Purpose | 본인과 친구들의 일일 달성률을 비교하여 순위를 매긴다. | 
| --- | --- | 
| Approach | 서버에서 나와 친구 관계로 묶인 사용자들의 당일 과제 달성률 데이터를 한 번에 조회해 가져온다. 과제 달성률을 기준으로 내림차순 정렬하여 화면에 순위 목록을 표시한다. | 
| Dynamics | 친구들과의 경쟁을 통해 현재 자신의 위치를 파악하고 분발하고 싶을 때.  | 
| Goals | 선의의 경쟁을 유도하여 서로의 과제 달성 의지를 극대화한다.|

# 5. Problem statement
아래는 프로젝트를 진행하기에 앞서 고려해야 할 기술적 어려움들에 대해 고찰한 결과이다.

## 5.1. Technical difficulties

**① 백그라운드 푸쉬 알림 구현**

Pomodoro Timer 기능이나 Daily Report, Reminder Task 기능을 구현하기 위해서는 푸쉬 알림을 보낼 수 있어야 한다. 사용자가 브라우저에 접속했을 때만 알림을 받을 수 있게 구현해도 되지만, 그러면 기능의 필요성이 낮아진다. 그러나 웹에서 알림을 보내는 기술을 공부해본 적이 없다는 것이 첫 번째 걸림돌이다.

**② 다양한 언어 사용으로 인한 개발 난이도**

프로그램 작성을 위한 언어, 서버 구현 외에도 알림 기능을 위한 서비스(FCM -Firebase Clout Messaging- 또는 비슷한 기능을 제공하는 다른 서비스)를 다시금 사용해야 한다. 이 정도 규모의 개발을 시도해 본 적도 없고, 이렇게 많은 언어와 개발 도구를 사용해보는 것도 처음이기에 개인의 능력으로 이 앱을 구현하려면 많은 노력을 쏟아야 할 것이다.

**③ 공동 과제 추가 중 서버 에러**

공동 과제는 서버가 관리한다. 그런데 공동 과제를 추가하던 중 서버 에러나 네트워크 문제가 발생한다면 일부는 과제를 받고, 일부는 받지 못하는 문제가 생긴다. 해당 문제를 막기 위해 과제 분배를 끝까지 진행하거나 전체 취소 후 다시금 시도하는 로직을 추가해야 할 것이다.

**④ 백그라운드 탭에서의 타이머 동기화**

Pomodoro Timer 기능에서 가장 중요한 것은 시간을 정확히 재고, 사용자가 기능을 멈추기 전까지 공부와 휴식 사이클을 반복하는 것이다. 그런데 백그라운드 상태에서 배터리 절약을 위해 타이머 주기가 느려지거나 멈춰버리는 사태가 발생하면 앱 사용에 불편함을 느끼게 될 것이다. 이러한 문제를 해결하기 위해 단순 시간 계산 함수를 사용하는 것이 아닌, 현재 시간과 시작 시점의 시간을 기록해 비교하는 방식을 사용하는 등 다른 방법을 모색해야 한다.

**⑤ 과제 추가 방식의 비효율성**

사용자가 앱을 이용하기 위해선 메인 과제만 입력하는 것이 아니라, 하위 과제들까지 하나하나 추가해야 한다. 이것이 귀찮은 일임은 명백하다. 비효율성을 감수하면서까지 앱을 사용할 이점을 제공하거나, 근본적으로 해당 문제점을 고칠 수 있는 방법을 찾아야 한다.
<br />
<br />

## 5.2. Non-Functional Requirements(NFRs)
다음은 프로그램이 ‘어떻게’ 수행되어야 바람직한지에 관한 고찰이다.

**① 응답성**
- Manage Task 기능으로 체크박스 형식의 과제 상태를 변경할 때 사용자에게 즉각적인 화면 변화를 제공해야 한다. 최대 0.5초 이내에 화면 변화를 작동하도록 하는 것이 바람직하다.
- View Ranking이나 View Status 기능으로 기록을 확인할 때 역시 오랜 로딩 시간을 거치지 않도록 주의해야 한다.

**② 신뢰성**
- Reminder Task와 Pomodoro Timer와 같이 현재 시간을 참조하는 기능에서, 정확한 시간에 알림이 발송될 수 있도록 해야 한다.
- 네트워크 지연이나 서버 오류로 인한 알림 유실률을 최소화할 수 있어야 한다.

**③ 보안**
- 본인의 과제 및 정보만 관리할 수 있고, 친구의 과제 내용은 수정하지 못하도록 권한을 엄격히 제한해야 한다.

# 6. Glossary
아래는 해당 보고서에 사용된 용어들의 구체적인 정의이다.



| Terms | Description |
| --- | --- | 
| 리더 (Leader) | 친구가 한 명 이상인 사용자. 공동 과제를 추가할 수 있는 권한을 부여받는다. | 
| 메인 과제 | 궁극적으로 달성하고자 하는 과제를 의미한다. 체크박스 형식으로 표시되는 하위 과제와 달리, 메인 과제는 체크박스 형식이 아니라 줄글 형식이며 하위 과제를 모두 완수해야만 해당 과제가 완료된 것으로 처리된다.  | 
| 하위 과제 | 궁극적으로 달성해야 하는 메인 과제를 세분화한 것으로, 체크박스 형식으로 나타낸다. 예를 들어, ‘보고서 10p 작성’이 목표라면 ‘노트북 켜기’, ‘관련 논문 찾아보기’, ‘초안 작성하기’ 등이 하위 과제가 될 수 있다. | 
| 공동 과제 | 리더 (Leader) 가 과제를 추가한 후 친구들과 공유할 수 있는 과제를 의미한다. 과제 추가 후 공동 과제로 등록하면, 서버가 해당 사용자의 친구 목록을 조회한 후 각 친구들에게 해당 과제를 자동 복사 후 추가한다. 이렇게 저장된 공동 과제는 개개인이 자신의 화면에서 독립적으로 관리할 수 있다. |
| 과제 달성률 | 오늘 생성된 과제 수와 완료 수의 비율이다. 메인 과제와 하위 과제의 가중치를 다르게 부여할 예정이며, 메인 과제의 가중치가 더 큰 값을 가진다. | 
| 연속 달성일 | 연속으로 과제 달성률이 100%인 날짜 수를 의미한다. 예를 들어, 5일간 과제 달성률이 100%였다면 해당 사용자의 연속 달성일은 5일이다. 연속 달성일을 이어가다가 과제 달성률이 100% 미만이 되면 연속 달성일은 0일로 리셋된다. |
| 연속 달성 격려 메시지 | 연속 달성일이 이틀 이상인 사람에게 발송되는 푸쉬 알림으로, 사용자의 성취감을 고취시키기 위해 고안되었다. |
| 푸쉬 알림 | 웹페이지에서 사용자에게 보내는 알림이다. 당일 과제 달성률을 보고하거나 연속 달성 격려 메시지를 보낼 때 사용된다. |
| 과제 리마인더 | 설정한 주기마다 미완료 상태인 하위 과제 수행을 촉구하는 푸쉬 알림이다. |
| 랭킹 | 친구들과 나의 과제 달성률을 계산해 내림차순 정렬 후 매긴 순위로, 친구를 추가한 사용자들이 이용할 수 있는 기능이다. |
